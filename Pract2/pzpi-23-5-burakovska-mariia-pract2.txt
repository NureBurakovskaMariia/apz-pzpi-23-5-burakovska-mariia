     3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони систему
     Назва системи: Steam.
     Розробник: Valve Corporation.
     Призначення: Хмарна платформа для цифрової дистрибуції відеоігор та програмного забезпечення, що забезпечує соціальну взаємодію користувачів (чати, спільноти), хмарні збереження та надає інструментарій API для розробників (Steamworks).
     Користувачі: Понад 120 мільйонів активних користувачів щомісяця з піковим навантаженням понад 30 млн одночасних підключень.

3.2  Архітектурний стиль системи
   
     Стиль: Гібридна сервіс-орієнтована архітектура (SOA) з елементами мікросервісів та глобальною мережею доставки контенту (CDN).
     Такий підхід забезпечує екстремальну стійкість до навантажень, особливо під час масштабних подій (розпродажів). Використання мікросервісів дозволяє незалежно масштабувати окремі функції: магазин, систему інвентарю, чати та сервіси завантаження файлів.
     
     3.3 Основні компоненти системи
      
     Інтерфейс користувача: Steam Client (Frontend), побудований на базі Chromium Embedded Framework (CEF) для відображення вебконтенту та власного UI-фреймворку бібліотеки ігор.
     Серверна частина: Connection Managers (CM) — сервери-посередники, що підтримують постійні TCP/UDP з'єднання та відповідають за маршрутизацію повідомлень.
     Рівень бізнес-логіки: Набір розподілених сервісів (Identity, Store, Inventory), що реалізують основні функції платформи.
     Рівень зберігання даних: Гібридна система, що поєднує SQL-бази для транзакційних даних та NoSQL/In-memory рішення (Redis) для кешування та швидких операцій із сесіями.
     Допоміжні сервіси: Steamworks Web API для взаємодії із зовнішніми іграми та Steam CDN для зберігання і передачі великих обсягів ігрових даних.
     
     3.4 Взаємодія компонентів
     
     Взаємодія базується на власному бінарному протоколі та використанні Protocol Buffers (Protobuf) для ефективної серіалізації даних.
     Потоки даних: Процес починається з підключення до CM, автентифікації через сервіс Identity та отримання персоналізованого списку ігор.
     Механізми синхронізації: Steam Cloud забезпечує синхронізацію файлів збережень між різними пристроями користувача через виділені сервіси зберігання.
     На рисунку 3.1 представлена діаграма послідовності, що описує сценарій авторизації та завантаження ігрового контенту. Клієнт отримує адресу найближчого контент-сервера від CM та завантажує дані окремими пакетами.
     Фізичний аспект взаємодії та розподіл компонентів по вузлах системи представлено на діаграмі розгортання (Рисунок 3.2). Архітектура передбачає взаємодію між пристроями користувачів (PC/Steam Deck), регіональними дата-центрами Valve та мережею серверів CDN Edge. Внутрішня комунікація між мікросервісами (Store, Identity, Inventory) у кластерах здійснюється за допомогою протоколів gRPC та Internal API, а робота з рівнем зберігання даних забезпечується через специфічні протоколи SQL та Redis.

     3.5 Масштабованість та надійність
     
     Балансування: Використовується DNS-балансування та інтелектуальний вибір Content Server на основі геолокації.
     Відмовостійкість: Мережа Steam Datagram Relay (SDR) захищає від DDoS-атак та оптимізує маршрути трафіку для зниження затримок.
     Кешування: Агресивне використання Edge-серверів для статичних сторінок магазину.
     Контейнеризація: Valve використовує контейнери для швидкого розгортання та масштабування серверної інфраструктури у глобальних дата-центрах.

     3.6 Приклади програмної реалізації

     Для підготовки матеріалів доповіді та аналізу архітектури Steam було використано сервіси штучного інтелекту.
     Назва інструменту та версія: Gemini 3 Flash.
     Мета використання: Збір та аналіз інформації щодо архітектурних підходів платформи Steam, структурування технічного звіту за вимогами дисципліни, розробка логічної схеми для UML-діаграми послідовності та підготовка розгорнутого тексту доповіді з поясненням складних технічних термінів.
     Внесок автора: Автор самостійно розробив стратегію запитів для отримання достовірних технічних даних, провів верифікацію наданої інформації на відповідність офіційній документації Valve та фактичним показникам масштабу системи (кількість користувачів та пікові навантаження). На основі отриманої логіки взаємодії компонентів автор власноруч створив графічне представлення UML-діаграми послідовності та адаптував усі згенеровані матеріали під вимоги, забезпечивши науково-технічний стиль викладу.
     Ключові запити розміщені у Додатку В.
     Для демонстрації архітектурного принципу відкритості та взаємодії через сервіси (Web API) в Додатку В також наведено приклад стандартного HTTP-запиту до інтерфейсу IsteamNews з офіційної документації. Це дозволяє зовнішнім системам отримувати дані з екосистеми Steam без прямого доступу до бази даних .

4 ВИСНОВКИ
     У результаті виконання практичного заняття було досягнуто поставленої мети та виконано всі завдання щодо аналізу архітектури складної програмної системи. Досліджено архітектуру платформи Steam, яка є прикладом високонавантаженої гібридної системи, що поєднує сервіс-орієнтований підхід (SOA) та мікросервісну модель. Визначено архітектурний стиль системи як такий, що базується на незалежному масштабуванні сервісів (магазин, інвентар, чат) та використанні глобальної мережі доставки контенту (CDN). Проаналізовано взаємодію компонентів і встановлено, що ключем до стабільності платформи при підтримці мільйонів активних сесій є використання мережі Connection Managers. Оцінено переваги архітектурних рішень, зокрема технології Steam Datagram Relay (SDR), яка дозволяє ефективно вирішувати проблеми мережевих затримок та захисту від DDoS-атак у глобальному масштабі. Встановлено, що гнучкість та масштабованість архітектури Steam є критично важливими факторами для підтримки високої працездатності системи в умовах сучасного ігрового ринку.
